## Slide 1: 整体技术流程

### 四个步骤分别做什么？

```
多视角图像 → VGGT(位姿+点云) → BA(优化位姿) → 3DGS(外观重建) → 渲染
```

**Step 1（VGGT）**：把 16 张没有相机标定的照片扔进一个深度学习模型，它一次性输出相机位置（外参）、镜头参数（内参）、每像素的深度、以及 3D 点云。**不需要传统 SfM 那种特征匹配+增量重建流程。**

**Step 2（BA）**：VGGT 的位姿是神经网络"猜"出来的，不够精确。BA 用经典的几何优化方法（SIFT 特征匹配 + 最小化重投影误差）把相机位姿和 3D 点坐标同时修正得更准。

**Step 3（3DGS）**：点云是一堆离散的点，中间有缝隙。3DGS 把每个点变成一个"半透明椭球"（Gaussian），然后通过可微渲染训练，让这些椭球学会互相配合，最终能从任意视角渲染出逼真的图像。

**Step 4（渲染）**：把训练好的 Gaussians 加载到 viewer 里实时看。

### 关键代码对应

| 步骤 | 文件 | 核心依赖 |
|------|------|---------|
| VGGT | `vggt_inference.py` | `VGGT.from_pretrained()`, `pose_encoding_to_extri_intri()` |
| BA | `bundle_adjustment.py` | `cv2.SIFT_create()`, `torch.optim.Adam` + `LBFGS` |
| 3DGS | `gsplat_training.py` | `gsplat.rasterization`, `gsplat.DefaultStrategy` |
| 渲染 | `gsplat_viewer.py`, `gradio_viewer.py` | `viser`, `gradio` |

## Slide 2: Step 1 — VGGT 求相机参数和初步点云

### 模型架构

VGGT-1B = **ViT 编码器**（Vision Transformer）+ **DPT 解码器**（Dense Prediction Transformer）。

- **ViT 编码器**：把 518×518 的输入图像切成 14×14=196 个 patch，每个 patch 经过 Transformer 自注意力，输出全局+局部特征。代码位置：`vggt/vggt/models/vggt.py` 中的 `VGGT` 类。
- **DPT 解码器**：把 ViT 的多层特征上采样回原图分辨率，分别输出位姿编码、深度图、点云。代码位置：`vggt/vggt/heads/` 目录。

### 推理流程（`vggt_inference.py`）

```python
# 1. 加载图像 (第 30-33 行)
from vggt.utils.load_fn import load_and_preprocess_images
images = load_and_preprocess_images(image_paths).to(device)
# 输出: (S, 3, 518, 518) — S帧, RGB三通道, 固定518分辨率

# 2. VGGT 前向推理
with torch.no_grad():
    predictions = model(images)
# predictions 包含: pose_enc, depth, depth_conf, world_points, world_points_conf

# 3. 位姿编码 → 相机矩阵 (第 96 行)
from vggt.utils.pose_enc import pose_encoding_to_extri_intri
extrinsic, intrinsic = pose_encoding_to_extri_intri(predictions["pose_enc"], images.shape[-2:])
# pose_enc 是 (S, 9) 的向量 — 前3个是translation, 中间4个是quaternion, 最后2个是FoV
# 解码后: extrinsic (S, 3, 4) world-to-camera, intrinsic (S, 3, 3)

# 4. 深度图反投影为世界坐标点云
from vggt.utils.geometry import unproject_depth_map_to_point_map
world_points = unproject_depth_map_to_point_map(depth_map, extrinsic, intrinsic)
```

### 关键细节

- **mask 过滤**：`apply_mask_filter()` 函数（`vggt_inference.py`）把 `msk_*.png` 中背景区域的 `depth_conf` 置零。这样后续采样点云时只取前景（人体）。
- **视频抽帧**：`extract_video_frames()` 函数用 `cv2.VideoCapture` 读视频，`np.linspace` 均匀采样指定帧数。如果帧被提取过（manifest.json 匹配），会跳过重复提取。
- **VGGT 只支持 518px 分辨率**：`load_and_preprocess_images` 会自动 resize。

### PPT 中的数字含义

- 深度范围 [0.37, 1.53]、均值 0.84：这是 VGGT 输出深度图的统计。深度值代表相机到物体表面的距离（归一化单位），1.0 大约对应场景尺度的一半。
- `depth_conf` 不是 [0,1] 的置信度，而是 VGGT 内部的置信度分数，值可以 >1（最高 139.77）。

## Slide 3: Step 2 — Bundle Adjustment

### 什么是 Bundle Adjustment？

BA 是计算机视觉中的经典优化问题。核心思想：

> 已知：多张照片中看到了一些相同的 3D 点
> 调整：相机位置 + 3D 点坐标
> 目标：让 3D 点投影到照片上的位置，尽可能接近实际观测到的像素位置

数学上：

$$\min_{R_i, t_i, X_j} \sum_{i,j} \rho\left( \| \pi(R_i X_j + t_i) - x_{ij} \|^2 \right)$$

- $R_i, t_i$：第 i 帧相机的旋转和平移
- $X_j$：第 j 个 3D 地标点
- $\pi$：投影函数（3D → 2D 像素）
- $x_{ij}$：在第 i 帧中观测到的第 j 个点的像素坐标
- $\rho$：Huber 损失（对离群点鲁棒）

### 实现详解（`bundle_adjustment.py`）

**① SIFT 特征提取（第 48 行）**

```python
sift = cv2.SIFT_create(nfeatures=2000)  # 每帧最多提取2000个关键点
kp, des = sift.detectAndCompute(gray_img, None)
# kp: 关键点列表（包含坐标、尺度、方向）
# des: 描述子 (N, 128) — 每个关键点对应一个128维浮点向量
```

SIFT 是怎么工作的？它找图像中"特别"的点（角点、斑点），这些点在旋转、缩放、光照变化下都稳定。每个点生成一个 128 维描述子，两个点的描述子距离越小，说明它们越可能是同一个物理点。

**② 特征匹配（第 72 行）**

```python
bf = cv2.BFMatcher(cv2.NORM_L2, crossCheck=True)
raw_matches = bf.match(des_i, des_j)
# 对上一帧的每个关键点，在下一帧中找描述子最相似的点
# crossCheck=True: 双向验证（A匹配到B，且B匹配到A，才算匹配）
```

**③ 三角化（第 104 行）**

```python
pt3d = cv2.triangulatePoints(P1, P2, pts_i[m], pts_j[m])
# 已知两个相机的投影矩阵 P1, P2 和匹配点对
# 求解两条射线的交点（最小二乘）
# 输出: 齐次坐标 (4,)，除以第4个分量得 (x,y,z)
```

**④ 优化变量参数化（第 230-265 行）**

```python
class BundleAdjustment(nn.Module):
    def __init__(self, ...):
        # 相机位姿参数化为 SE3 李代数 (6维向量)
        # 前3维 = 旋转向量 (axis-angle), 后3维 = 平移
        se3_init = extrinsics_to_se3(extrinsics_init)  # (S, 3, 4) → (S, 4, 4)
        pose_init = se3_log(se3_init)  # (S, 4, 4) → (S, 6)
        self.pose_params = nn.Parameter(torch.from_numpy(pose_init).float())

        # 3D地标点直接作为可优化变量
        self.points3d = nn.Parameter(torch.from_numpy(points3d_init).float())
```

**为什么用 SE3 而不是直接用 3×4 矩阵？** 直接用 3×4 矩阵（12 个参数）会优化出非法的旋转矩阵（不再是正交矩阵）。SE3 只有 6 个自由度，保证旋转矩阵始终合法。

**⑤ SE3 指数映射（第 280-320 行，`se3_exp_torch` 函数）**

```python

```

这是你自己实现的（PPT 说"自写 Rodrigues 公式，autograd 兼容"指的就是这个）。相比调 `scipy.spatial.transform.Rotation`，用纯 PyTorch 写的好处是梯度可以自动回传。

**⑥ 两阶段优化**

```python
# 阶段1: Adam，500步，学习率从0.001开始，每200步减半
optimizer = torch.optim.Adam(ba.parameters(), lr=0.001)
scheduler = torch.optim.lr_scheduler.StepLR(optimizer, step_size=200, gamma=0.5)

# 阶段2: L-BFGS，20步精调
lbfgs_optimizer = torch.optim.LBFGS(ba.parameters(), lr=0.01, max_iter=20,
                                     line_search_fn="strong_wolfe")
```

Adam 负责快速收敛到大方向，L-BFGS（一种二阶优化方法）负责精细调优。L-BFGS 利用曲率信息（近似的 Hessian 矩阵），在接近最优解时比 Adam 更精确。

**⑦ Huber Loss**

```python
self.huber = nn.HuberLoss(delta=1.0)
```

Huber 损失在误差小时等于 MSE（平方），误差大时等于 MAE（线性）。作用：个别匹配错误的点（离群点，误差很大）不会主导梯度方向。δ=1.0 代表误差在 1 像素以内时用平方惩罚，超过 1 像素用线性惩罚。

## Slide 4: Step 3 — 3D Gaussian Splatting

### 什么是 3D Gaussian？

一个 3D Gaussian 是一个**三维空间中的椭球**，用 5 个属性描述：

| 属性 | 符号 | 维度 | 含义 |
|------|------|------|------|
| 位置 | μ | (3,) | 椭球中心 |
| 协方差 | Σ | (3,3) | 椭球的形状和朝向（分解为 scale×rotation） |
| 不透明度 | α | (1,) | 0=完全透明, 1=完全不透明 |
| 颜色 | c | (3,) 或 (16,3) | RGB 颜色（用球谐系数表示，支持视角相关颜色） |

### 为什么叫 Splatting？

"Splat" = 把一个 3D Gaussian **投影到 2D 屏幕上**。数学上：

- 3D Gaussian：$G(x) = e^{-\frac{1}{2}(x-\mu)^T \Sigma^{-1} (x-\mu)}$
- 投影到屏幕后还是 Gaussian（近似），中心在投影点，协方差用相机矩阵变换

### 训练过程（`gsplat_training.py`）

**① 初始化（`load_pipeline_data` 函数，第 148 行）**

```
VGGT深度图 + BA位姿 → 反投影3D点 → 置信度过滤 → 采样150k个点
→ 位置=点坐标, 颜色=像素RGB, 不透明度=置信度映射, 缩放=KNN估计
```

初始化的关键：
- **位置直接取点云坐标**，不做修改
- **颜色从对应像素取 RGB**，然后转换为球谐系数的 DC 分量（`rgb_to_sh` 函数，第 80 行）
- **缩放用 KNN 估计**：找每个点最近的几个邻居，取平均距离作为初始半径。代码中把缩放存在 log 空间（`torch.log(scales)`），因为缩放必须 >0，log 参数化后可以无约束优化

**② 可微光栅化（`rasterization` 调用）**

```python
from gsplat import rasterization
rendered, alpha, info = rasterization(
    means, quats, scales, opacities, colors,
    viewmats, Ks, width, height, ...
)
# 对每个像素，收集所有覆盖到的 Gaussians
# 按深度排序，alpha-blend 得到像素颜色
# 整个过程是 CUDA kernel，支持自动求导
```

光栅化的核心是 **alpha-blending（α 混合）**：

$$C(pixel) = \sum_{i} c_i \cdot \alpha_i \cdot \prod_{j=1}^{i-1} (1-\alpha_j)$$

从最近到最远逐个混合 Gaussian 的颜色，后面的 Gaussian 会被前面的遮挡。

**③ DefaultStrategy 增密/裁剪**

PPT 中的 densify window [500, 3500] 意思：
- 第 500 步开始，每隔一定步数检查
- 如果某个 Gaussian 的梯度累积大（说明它收到的监督信号强、还需要"生长"），就分裂成两个
- 如果某个 Gaussian 透明度太低或太大，就移除或拆分
- 第 3500 步后停止增密，后面只优化

这是 `gsplat.DefaultStrategy` 自动处理的，不需要手动写。代码在 `gsplat_training.py` 的训练循环中调用。

**④ 损失函数**

```python
loss = 0.8 * L1(pred, target) + 0.2 * (1 - SSIM(pred, target))
```

- **L1**：逐像素绝对值误差，保持边缘锐利
- **SSIM**：结构相似性，衡量感知质量（亮度+对比度+结构）
- **深度监督（可选）**：额外要求渲染的深度图和 VGGT 预测的深度一致

**⑤ 训练循环（`gsplat_training.py` 训练部分）**

```python
for step in pbar:
    cam_idx = np.random.randint(S)  # 随机选一帧
    pred = render(viewmat[cam_idx])
    loss = compute_loss(pred, gt_images[cam_idx])
    loss.backward()
    strategy.step_post_backward(params, ...)  # 增密/裁剪
    optimizer.step()
    optimizer.zero_grad()
```

每步随机选一个视角渲染，这是 3DGS 的标准训练方式。这样 Gaussians 必须学会从**所有视角**都好看，而不是只记住某一个视角。

## Slide 5-10: 各数据集重建结果

### 如何解读这些数据？

**人体数据（数据1、数据2）：**

- VGGT 位姿下 PSNR 高（38 dB），BA 后反而下降
- 但 BA 后 Gaussians 数量少（149k vs 166k, 95k vs 179k）

**这意味着什么？** BA 改了相机位置。虽然改动很小（平均平移变化 0.027），但对 3DGS 训练来说，相机位置一变 → 同一个像素从不同视角看到的东西不一样 → 3DGS 需要在"出厂默认设置"（VGGT 位姿）和"BA 修正设置"之间重新适应。7000 步可能不足以完全收敛到 BA 位姿下的最优。这就是为什么 BA 后 **Gaussians 更少但 PSNR 没更高** 的原因——Gaussians 还没来得及学会利用更准确的几何关系。

**场景数据（数据3）：**

- VGGT 位姿 PSNR 低（30.08 dB, 476k Gaussians）
- BA 后 PSNR 提升（33.64 dB, 363k Gaussians）

VGGT 在场景上给了很差的位姿（80 px 误差），3DGS 需要用很多 Gaussians 来"弥补"位姿错误（同一个墙在不同视角下位置对不上，Gaussians 需要长胖来覆盖）。BA 修正位姿后，Gaussians 只需要正常大小就能覆盖，所以数量减少 + PSNR 提升。

## Slide 12: BA 对 3DGS 效果的分析

### 核心概念：稀疏 BA vs 稠密 3DGS

这是 PPT 中最重要的分析，答辩时可能会被追问。

- **BA 优化目标**：让 SIFT 关键点（~1272 个）的重投影误差最小
- **3DGS 优化目标**：让**所有像素**（518×518×16 = 430 万个）的 L1+SSIM 误差最小

两者优化的不是同一个东西。BA 把稀疏关键点调对了，不代表所有像素的光度误差就变小了。极端情况：BA 可能为了调对某个角点而把相机稍微旋转了一下，结果旁边的大片平坦区域反而投影偏差变大。

### PSNR "反降"的真实原因

PPT 中提到人体数据 BA 后 PSNR 不升反降，这不是 bug。原因有三：

1. **PSNR 在训练视角上测**：3DGS 见过这些视角，可以"背下来"。VGGT 的位姿虽然有小误差，但 3DGS 可以用更多 Gaussians 来适应。BA "修正"视角后，3DGS 需要重新适应。

2. **BA 改变了像素对应关系**：BA 微调了相机位姿 → 同一帧图像现在映射到略微不同的 3D 位置 → Gaussians 之前的"记忆"失效 → PSNR 暂时下降。

3. **真正的收益在泛化能力上**：如果留出一帧不训练只测试（留出测试），BA 位姿通常在新视角上表现更好。因为 BA 保证了几何一致性——你做实验时没做留出测试，所以只能看到训练视角上的 PSNR。

### 答辩时如何回应质疑

如果有人问"BA 反而降低了 PSNR，那 BA 是不是没用？"：

> BA 改善的是几何一致性，不是训练视角的 PSNR。证据是 BA 后 Gaussians 减少了 10-47%——如果位姿不好，3DGS 需要用更多 Gaussians 来补偿不一致性。BA 的真正价值应该在新视角上评估，这受限于当前数据集没有留出测试帧。

## Slide 13: VGGT 改进方法

### Mask 引导置信度过滤（已实现）

PPT 中"前景仅 8.2%"的意思：16 张 518×518 的人体图像，只有 8.2% 的像素属于人体（mask 白色区域），其余 91.8% 是背景。过滤后这些背景像素的 depth_conf=0 → 3DGS 初始 Gaussians 全部集中在人体上 → 背景不会占用 Gaussians 配额。

**实现**：`vggt_inference.py` 中 `apply_mask_filter()` → 加载 `msk_*.png` → resize 到模型分辨率 → 背景区域的 `depth_conf` × 0。

### Depth+Pointmap 多分支融合（已实验）

VGGT 有两个输出路径：
1. **Depth Branch**：预测深度图 → 反投影得到 3D 坐标。平面区域平滑。
2. **Pointmap Branch**：直接输出 3D 世界坐标。细节处更准。

融合公式：
```python
alpha = depth_conf / (depth_conf + pointmap_conf)  # 逐像素权重
world_points_fused = alpha × world_points_from_depth + (1-alpha) × world_points_pointmap
```

实验结果（数据1-人体, VGGT 位姿）：融合后 α 均值仅 0.05 → Pointmap 分支贡献 95%。因为 mask 过滤后深度分支前景置信度低，Pointmap 分支在人体上置信度高。PSNR 仅 +0.19 dB → 边际收益，因为两个分支在前景区域本身就很一致。

## Slide 14: 未来方向

### 端到端联合优化

当前流程：VGGT → BA → 3DGS 是三个独立步骤，误差单向传递。如果 VGGT 输出的深度在某个区域差，3DGS 无法纠正。

理想做法：让 3DGS 的渲染损失反向传播到 VGGT 的深度/位姿估计，端到端训练。但 VGGT-1B 参数太多，目前 GPU 内存不够。

### 动态场景

VGGT 假设场景静止。如果画面里有人在走动，VGGT 会认为"这个物体的 3D 位置一直在变"→ 位姿估计混乱。解决方向：先估计哪些像素是动态的（光流/分割），训练时排除这些像素。

### 少视图重建

VGGT 需要至少 8 张视角差异够大的图。如果只有 2-3 张手机随手拍的照片（重叠很小），VGGT 可能失败。改进方向：加入单目深度先验（如 DepthAnything），用单张图预测的深度补充缺失的 3D 信息。

## 附录 A: PSNR 和 SSIM 怎么算的

### PSNR（Peak Signal-to-Noise Ratio）

$$PSNR = 20 \cdot \log_{10}\left(\frac{1.0}{\sqrt{\frac{1}{N}\sum_{i}(pred_i - gt_i)^2}}\right)$$

代码（`gsplat_training.py` 第 84 行）：
```python
def compute_psnr(pred, target):
    mse = F.mse_loss(pred, target)  # 均方误差
    return 20.0 * torch.log10(1.0 / torch.sqrt(mse))
```

假设像素值在 [0, 1]，MAX = 1.0。MSE=0.0001 → PSNR=40dB，MSE=0.001 → PSNR=30dB。每提升 6 dB 约等于误差减半。

### SSIM（Structural Similarity）

不像 PSNR 只看逐像素差异，SSIM 从三个维度衡量：
- **亮度**(luminance)：平均值是否一致
- **对比度**(contrast)：方差是否一致
- **结构**(structure)：归一化后的相关系数

代码（`gsplat_training.py` 第 58-76 行）：用高斯窗在 11×11 的局部块上计算，然后全图平均。值域 [0,1]，越接近 1 越像。

### 为什么用 0.8×L1 + 0.2×(1-SSIM)？

这是 3DGS 原论文的标准组合。L1 保持边缘锐利（L2 会让边缘模糊），SSIM 提升感知质量。0.2 的权重给了 SSIM 足够的影响力但不会主导梯度方向。

## 附录 B: 球谐函数（Spherical Harmonics, SH）

PPT 中提到 "颜色 = 像素 RGB → SH0"。球谐是用于表示**视角相关的颜色**的数学工具。

- SH degree=0（1 个系数）：表示从任何角度看颜色都一样的"基础颜色"（漫反射）
- SH degree=1（额外 3 个系数）：表示颜色的方向变化（如高光）
- SH degree=2（额外 5 个系数）：更精细的方向变化
- gsplat 默认用 degree=3，总共 16 个系数 × 3 个颜色通道 = 48 个参数

初始化时只设置 DC 分量（SH0），其他分量初始化为 0（各向同性颜色）。训练过程中高次系数会逐渐学到高光和视角相关的颜色变化。

## 附录 C: 答辩追问（22 个问题，按 PPT 顺序排列）

### 一、VGGT 相关

**Q1：VGGT 和 COLMAP 有什么区别？为什么选 VGGT？**

VGGT 是端到端深度学习，1.6s 出结果，COLMAP 是经典 SfM 流程，需要几十分钟。VGGT 学到了"3D 世界长什么样"的先验，对白墙、皮肤等无纹理区域也能预测合理深度，而 COLMAP 在纹理稀疏处直接失败。但 VGGT 的位姿精度不如 COLMAP——这就是为什么后面要加 BA。

**Q2：VGGT 输入 16 帧，如果只给 2-3 帧会怎样？**

VGGT 依赖多视角关系来推断 3D 结构。帧数太少 → Transformer 的跨帧注意力几乎没有信息 → 位姿和深度都不准。最低建议 8 帧，16 帧是稳定值。

**Q3：VGGT 输出的 pose_enc 是怎么解码成相机矩阵的？**

`pose_enc` 是 (S, 9) 的向量。每帧的 9 个数：前 3 个是相机光心在世界坐标系中的位置（translation），中间 4 个是旋转四元数 (w, x, y, z)，最后 2 个是视场角 FoV（用来算焦距）。解码函数 `pose_encoding_to_extri_intri()` 在 `vggt/vggt/utils/pose_enc.py`。

**Q4：VGGT 的深度置信度有时候 > 100（如 139.77），这合理吗？**

`depth_conf` 不是 [0,1] 的概率，而是 VGGT 内部未经归一化的置信度分数。值越大越确定，可以是任意正数。在 mask 过滤和 3DGS 初始化时，我们只关心相对大小（取 p30 阈值），不关心绝对值。

**Q5：VGGT 的人体初始重投影误差 15-19 px，算好还是差？**

算**偏差但可用**。COLMAP 能做到 <1 px，但那是几十万次特征匹配 + 多轮 BA 的结果。VGGT 一次前向推理就能到 ~15 px，对于直接初始化 3DGS 来说是合格的。场景 81 px 就太差了，所以 BA 在那里效果最明显。

### 二、BA 相关

**Q6：你的 BA 和经典 BA（如 COLMAP 用的）有什么不同？**

| | 我的 BA | 经典 BA (CERES/g2o) |
|------|---------|---------------------|
| 优化器 | Adam + L-BFGS | Gauss-Newton / Levenberg-Marquardt |
| 求导方式 | PyTorch autograd | 手写解析 Jacobian |
| 位姿参数化 | SE3 李代数 (6 维) | 同 |
| 鲁棒核 | Huber (δ=1) | Huber / Cauchy |
| 稀疏性利用 | 否（稠密求解） | 是（利用 Schur 补） |

我的实现 ≈500 行 Python，不依赖任何外部 BA 库。对于 ≤96 帧、≤1300 地标的规模，稠密求解完全够用（3 秒跑完）。

**Q7：为什么不用 Gauss-Newton 而用 Adam？**

Gauss-Newton 需要手写 Jacobian 矩阵（每个误差项对每个参数的偏导），对于 SE3 参数化和自定义的 Rodrigues 公式来说很繁琐。Adam + L-BFGS 用 PyTorch 的 autograd 自动算梯度，实现量大幅减少。代价是收敛稍慢，但 500 步 Adam 也只需要 ~2 秒。

**Q8：为什么 L-BFGS 经常失败（代码里有 try-except）？**

L-BFGS 对数值稳定性敏感。当梯度接近零或 Hessian 近似矩阵条件数差时，`strong_wolfe` 线搜索可能找不到合适的步长而崩溃。代码 catch 了这个异常，回退到 Adam 的结果——PPT 里的 BA 数据就是 Adam 的结果，L-BFGS 失败了没影响最终数值。

**Q9：BA 后的重投影误差（13-31 px）还是比经典 SfM（<1 px）大很多，为什么？**

三个原因：①只匹配了相邻帧（15 对），没有跨帧/回环匹配——每条射线只用两次，没有多余约束来"拉紧"；②只有 400-1300 个地标，经典 SfM 通常用数万个；③VGGT 的内参是预测的，固定不变——如果内参本身有偏差，外参和点的优化空间就被限制了。

**Q10：你的 BA 优化了内参吗？**

没有。`intrinsic` 在 `BundleAdjustment` 类中被 `register_buffer` 固定。选择不优化内参是因为 16 帧数据用同一相机（手机拍摄），内参真实值不变，VGGT 预测的内参也足够接近。

**Q11：为什么稀疏 BA 能改善稠密的 3DGS？两者的优化目标不一样啊？**

确实不一样。但 BA 改善的是**几何一致性**——相机之间的相对位置关系变得更准确。这对 3DGS 很重要：同一个 Gaussian 在相邻视角下投影到正确的位置，3DGS 才能收到一致的 RGB 监督。如果位姿不准，Gaussian 在不同视角下"对不上"，它只能靠增密/长胖来勉强覆盖。

### 三、3DGS 相关

**Q12：一个 3D Gaussian 有 59 个可学习参数，这么多不会过拟合吗？**

59 个参数 = 位置(3) + 旋转(4) + 缩放(3) + 不透明度(1) + 球谐颜色(16×3=48)。但训练时一张图有 ~27 万像素（518×518），每个 Gaussian 只覆盖一部分像素，所以观测远多于参数。而且 DefaultStrategy 的增密/裁剪机制会自动调节 Gaussians 数量，防止过拟合和欠拟合。

**Q13：gsplat 的光栅化是怎么做到可微的？**

CUDA kernel 遍历每个 tile（16×16 像素块），收集覆盖到该 tile 的所有 Gaussians，按深度排序，然后做 alpha-blending。关键在于**排序和混合都是连续的**——颜色是 Gaussians 颜色的加权和，权重是 opacity × 高斯值，高斯值是位置/协方差的连续函数，所以整个 pipeline 可以自动求导。

**Q14：为什么你说 human 数据上 "BA 后 PSNR 反而低" 不代表 BA 没用？**

因为 PSNR 在训练视角上测的。3DGS 见过训练视角，可以把 VGGT 的位姿误差"背下来"（用更多 Gaussians 补偿）。BA 改了位姿→像素对应关系乱了→3DGS 要从头学→7000 步来不及完全收敛→PSNR 暂时低。真正的测试是留出一帧不训练，用训练好的 Gaussians 渲染留出帧——这时 BA 位姿的优势才能体现。这是本项目的主要局限性之一。

**Q15：3DGS 渲染需要多少显存？FPS 多少？**

训练：15 万 Gaussians，约 4-6 GB 显存（视场景大小和背景复杂度）。渲染：gsplat 在 4090 上可达 100-200 FPS（取决于 Gaussians 数量）。交互查看器中因为包含 viser 的 WebSocket 传输开销，实际约 30-60 FPS。

**Q16：为什么场景数据的 Gaussians 长到 47 万，而人体只到 15-17 万？**

场景包含墙壁、地板、树叶、天空等大面积几何，这些区域虽然纹理不复杂，但需要很多 Gaussians 来覆盖连续的平面。人体只有一个人形区域，几何紧凑。另外 VGGT 位姿越差→3DGS 越需要增密来补偿→Gaussians 越多。

**Q17：为什么人体数据用 mask-black 合成黑底，场景保留原图？**

人体有 `msk_*.png`，背景明确。3DGS 如果将 Gaussians 浪费在背景上：①占用显存和训练时间；②背景在不同视角下的一致性不保证（绿幕/背景在不同帧中不同）。合成黑底让 3DGS 只学习人体。场景数据没有 mask，且背景是真实的室内环境——保留原图让 3DGS 重建完整空间，更有展示价值。

**Q18：你用的深度监督 (depth supervision) 是怎么起作用的？**

每一训练步除了计算 RGB 损失，还额外渲染一张深度图（从 Gaussians 的加权深度），和 VGGT 原始深度图做 L1 损失。权重 0.5。作用：引导 Gaussians 的深度和 VGGT 预测一致，防止 Gaussians 在缺乏纹理监督的区域（如纯色衣服）乱飘。场景数据中深度监督的阈值较高（p30 以上才参与），因为场景深度范围大，低置信度的深度反而会误导。

### 四、改进方法相关

**Q19：多分支融合的 α 均值只有 0.05，那融合和直接用 Pointmap 几乎没区别？**

对，在人体数据上几乎等价。因为 mask 过滤后深度分支的前景置信度偏低（平均 3.7），而 Pointmap 的置信度虽然绝对值更低（平均 1.2），但相对更稳定。在**没有 mask 过滤的场景数据**上，两个分支的置信度分布差异更大，融合收益预期更高。

**Q20：还有什么改进没做但值得做的？**

1. **留出测试**（最重要）：留 1-2 帧不参与 3DGS 训练，在新视角上测 PSNR。这样 BA 对泛化能力的提升才能被验证
2. **全局匹配 + 回环**：不只匹配相邻帧，还要匹配"看起来像"的非相邻帧（用词袋模型），加回环约束后 BA 效果会大幅提升
3. **联合优化 VGGT+3DGS**：让 3DGS 的渲染损失反向传播到 VGGT 的深度，但 VGGT 太大（1B），需要 LoRA 等轻量化方案

### 五、工程与实现相关

**Q21：整个项目的代码量是多少？从零开始写的吗？**

核心 pipeline 约 2700 行 Python（8 个文件）。VGGT 模型本身是 Meta 发布的，BA 是纯自己写的（~500 行），3DGS 训练借用了 gsplat 的光栅化和 DefaultStrategy 但初始化、数据加载、训练循环、导出都是自己写的。前后调试/修改过至少 20+ 次。

**Q22：4090 够用吗？如果只有 8GB 显存呢？**

4090 (24GB) 完全够用。8GB 情况下：VGGT 推理 OK（~6GB），3DGS 训练需要降低 `--max-points` 到 50000 左右，或减少输入帧数到 8-12 帧。场景数据 Gaussians 可能突破显存限制（47 万 ~ 10GB），需要调低增密上限。
